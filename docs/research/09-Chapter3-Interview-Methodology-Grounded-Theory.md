## 1. 访谈研究设计

### 1.1 研究范式

| 维度 | 选择 | 理由 |
|------|------|------|
| **研究范式** | 解释主义 (Interpretivism) | 理解用户主观体验和意义建构 |
| **方法论** | 扎根理论 (Grounded Theory) | 从数据中归纳理论，适合探索性研究 |
| **数据收集** | 半结构化深度访谈 | 平衡结构性与灵活性 |
| **抽样策略** | 理论抽样 (Theoretical Sampling) | 根据涌现概念指导后续抽样 |

### 1.2 参与者招募

**招募渠道:**
- 新加坡南洋理工大学 (NTU) 学生邮件列表
- 厦门大学、清华大学、人民大学研究生群
- LinkedIn专业人士网络
- 滚雪球推荐 (参与者推荐其他潜在参与者)

**纳入标准:**
1. 年龄≥18岁
2. 过去6个月内使用过生成式AI工具（ChatGPT、Claude、Gemini等）
3. 使用频率≥每周1次
4. 愿意参与45-90分钟深度访谈
5. 能够用中文或英文流利交流

**排除标准:**
1. AI/ML领域的核心开发者（可能有专业偏见）
2. 仅使用AI进行娱乐目的（非学习/工作情境）

### 1.3 样本规模决策

**初始目标:** 30-50人（基于质性研究惯例）

**实际样本:** N=49

**样本规模理由:**
- 在第35次访谈后观察到理论饱和迹象
- 继续至第49次以确认饱和并增加模式多样性
- 最终六模式分布稳定，无新模式涌现

### 1.4 访谈实施

| 参数 | 详情 |
|------|------|
| **时间范围** | 2024年3月-2024年8月（6个月） |
| **访谈时长** | 45-93分钟（平均62分钟） |
| **访谈形式** | Zoom视频通话（录制） |
| **访谈语言** | 中文41人，英文8人 |
| **转录方式** | 自动转录 + 人工校对 |
| **总转录字数** | 约487,000字 |

## 2. 半结构化访谈协议

### 2.1 访谈指南 (Interview Guide)

**开场 (5分钟)**

```
感谢您参与本研究。本访谈旨在了解您使用生成式AI工具（如ChatGPT）的
体验和策略。没有对错之分，我们只是想了解您的真实想法和做法。

访谈将持续约60分钟，全程录音用于后续分析。您的信息将严格保密，
研究报告中使用匿名编号。您可以随时暂停或终止访谈。

开始前，您有任何问题吗？
```

**Part A: 背景与使用概况 (10分钟)**

| 编号 | 问题 | 追问提示 |
|------|------|---------|
| A1 | 请简单介绍一下您自己——目前的学习/工作情况？ | 专业、年级/职位、主要任务 |
| A2 | 您最常使用哪些AI工具？使用频率如何？ | 具体工具名称、每周使用天数 |
| A3 | 您主要用AI做什么类型的任务？ | 学习、工作、创作、编程等 |
| A4 | 您大约从什么时候开始使用这类工具的？ | 使用时长、初次接触契机 |

**Part B: 具体使用场景深挖 (25分钟)**

| 编号 | 问题 | 追问提示 |
|------|------|---------|
| B1 | 能否描述一个您**最近**使用AI完成的具体任务？ | 任务背景、目标、过程、结果 |
| B2 | 在开始这个任务前，您做了哪些准备或规划？ | 是否分解任务、设定目标 |
| B3 | 您是如何与AI交互的？能否回忆具体的对话过程？ | Prompt策略、迭代次数 |
| B4 | 您如何判断AI的输出是否满足您的需求？ | 验证方法、质量标准 |
| B5 | 这个任务中，哪些部分您自己做，哪些交给AI？ | 分工逻辑、边界意识 |
| B6 | 如果AI的输出不符合预期，您通常怎么处理？ | 迭代策略、放弃阈值 |
| B7 | 能否再描述一个**不太成功**的AI使用经历？ | 失败原因、学到什么 |

**Part C: 元认知与反思 (15分钟)**

| 编号 | 问题 | 追问提示 |
|------|------|---------|
| C1 | 您觉得自己对AI的信任程度如何？这种信任有变化吗？ | 信任来源、信任演化 |
| C2 | 使用AI后，您觉得自己的能力有什么变化？ | 技能提升/退化、学习效果 |
| C3 | 您有没有担心过过度依赖AI？如果有，您怎么应对？ | 依赖意识、保护策略 |
| C4 | 您认为AI在哪些任务上表现好？哪些不好？ | 任务类型认知、局限性理解 |
| C5 | 如果要给其他人建议如何有效使用AI，您会说什么？ | 元策略、经验总结 |

**Part D: 未来展望 (5分钟)**

| 编号 | 问题 | 追问提示 |
|------|------|---------|
| D1 | 您未来打算如何使用AI？有什么计划或担忧？ | 使用意向、预期变化 |
| D2 | 您觉得学校/公司应该如何引导AI的使用？ | 政策建议、培训需求 |

**收尾 (5分钟)**

```
非常感谢您的分享！在结束前，有没有什么我们没聊到但您觉得重要的内容？
您对本研究有什么建议或问题吗？

[发送礼品卡/感谢邮件]
```

### 2.2 追问技巧

**深挖技巧 (Probing Techniques):**

| 技巧 | 示例 | 使用场景 |
|------|------|---------|
| **具体化** | "能举个具体例子吗？" | 回答过于抽象 |
| **澄清** | "您说的'验证'具体是指什么？" | 术语模糊 |
| **对比** | "这和之前那个任务有什么不同？" | 探索情境差异 |
| **假设** | "如果AI给了错误答案，您会怎么办？" | 探索应对策略 |
| **反思** | "回顾这个经历，您有什么感想？" | 引导元认知 |
| **沉默** | [等待5-10秒] | 给予思考空间 |

## 3. 扎根理论编码过程

### 3.1 编码流程概述

```
┌─────────────────────────────────────────────────────────────┐
│                    扎根理论编码三阶段                         │
│              (Strauss & Corbin, 1998)                       │
└─────────────────────────────────────────────────────────────┘

Phase 1: 开放编码 (Open Coding)
├── 逐行分析访谈文本
├── 标注初始概念 (Initial Concepts)
├── 生成587个初始代码
└── 产出: 初始代码本 (Initial Codebook)

                    ↓

Phase 2: 轴心编码 (Axial Coding)
├── 建立代码间关系
├── 识别核心类别 (Core Categories)
├── 整合为12个子过程 (P1-P4, M1-M3, E1-E3, R1-R2)
└── 产出: 12维元认知框架

                    ↓

Phase 3: 选择性编码 (Selective Coding)
├── 围绕核心现象整合
├── 识别六种使用模式 (Pattern A-F)
├── 建立模式判别规则
└── 产出: 六模式分类理论
```

### 3.4 选择性编码详细过程

**核心类别识别:**

通过持续比较分析，识别出**"元认知调节强度"**作为核心类别，它决定了用户与AI交互的整体模式。

**模式判别维度确定:**

| 维度 | 高表现特征 | 低表现特征 |
|------|-----------|-----------|
| **规划完整性** | 系统分解、明确目标 | 无规划、模糊目标 |
| **监控主动性** | 持续追踪、主动校准 | 被动接受、无监控 |
| **评价批判性** | 深度验证、风险意识 | 零验证、盲目信任 |
| **调节灵活性** | 策略迭代、工具切换 | 固定模式、无调整 |

**六模式涌现:**

通过对49名参与者在四个维度上的表现进行聚类分析，涌现出六种典型模式：

| 模式 | 规划 | 监控 | 评价 | 调节 | 典型特征 |
|------|------|------|------|------|---------|
| **A** | 高 | 中 | 高 | 中 | 战略性分解与控制 |
| **B** | 中 | 高 | 中 | 高 | 迭代优化与校准 |
| **C** | 中 | 中 | 中 | 中 | 情境敏感的适配 |
| **D** | 中 | 高 | 高 | 中 | 深度核验与批判性介入 |
| **E** | 高 | 高 | 高 | 高 | 教学化反思与自我监控 |
| **F** | 低 | 低 | 低 | 低 | 元认知缺位与被动依赖 |

## 6. 六模式涌现叙事

### 6.1 模式涌现时间线

**Phase 1: 初始模式识别 (I1-I10)**

在前10次访谈中，我们注意到用户AI使用策略的显著差异：

> **I1 (后分类为模式A)**: "我自己会一段一段地做，先列大纲..."
> **I7 (后分类为模式B)**: "我需要看到第一个输出才知道下一步怎么走..."
> **I4 (后分类为模式C)**: "要看情况，简单任务直接问，复杂任务会多想想..."

**初步假设:** 存在"高规划型"、"迭代型"、"情境型"三种模式。

**Phase 2: 模式细化 (I11-I20)**

继续访谈后，发现"高规划型"内部存在分化：

> **I16 (后分类为模式D)**: "我会用两个AI做左右脑互补...然后我自己做最终决策"
> **I13 (模式A)**: "我有明确的分工，核心论点自己写，背景信息让AI帮忙"

**关键洞察:** 验证深度是区分模式A和模式D的关键。模式D用户的验证行为更加系统化和多工具化。

**Phase 3: 极端模式发现 (I21-I30)**

> **I6 (模式E)**: "我把每次和AI的对话当作学习机会，会记录什么有效什么无效..."
> **I30 (模式F)**: "基本上就是复制粘贴，没什么特别的...它给什么我就用什么"

**关键发现:**
- 模式E代表"元认知顶峰"——将AI交互视为学习过程
- 模式F代表"元认知缺位"——被动接受AI输出

**Phase 4: 模式验证 (I31-I49)**

后续访谈未发现新模式，但丰富了各模式的变体：
- 模式A的"时间压力变体"（I34疲劳滑坡现象）
- 模式C的"任务敏感变体"（不同任务不同策略）

### 6.2 模式命名理据

| 模式 | 命名 | 理据 |
|------|------|------|
| **A** | Strategic Decomposition & Control | 核心特征是战略性任务分解和边界控制 |
